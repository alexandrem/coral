# RFD 105 - ServiceStatus Proto Extension

**Status:** 🚧 Draft

## Summary

Extend `ServiceStatus` in `agent.proto` to expose the unified service map
introduced by RFD 104. Add a `ServiceNamingSource` enum, auto/authoritative
name fields, observation tier, and monitor state. Update the `ListServices` RPC
handler to merge auto-observed and watched entries into a single response.

## Problem

**Current behavior/limitations:**

`ListServices` only returns services that were explicitly registered via
`coral connect`. Auto-observed services (from default-on observation, RFD 104)
have no proto representation and cannot be queried over gRPC.

**Why this matters:**

- `coral services` (RFD 107) needs to display auto-observed services; without
  a proto surface it cannot retrieve them from a remote agent.
- The colony's `ListServices` aggregation has no way to distinguish an
  auto-observed process from one the operator has explicitly named.
- AI and CLI tooling cannot tell whether a service name was user-provided or
  auto-generated.

## Solution

Extend `ServiceStatus` with the fields from `ServiceEntry` (RFD 104) and
update `ListServices` to include all entries regardless of naming source.

**Key Design Decisions:**

- **Additive-only proto change.** All new fields use previously unassigned
  field numbers; existing fields are untouched. Old CLI versions reading the
  response silently ignore new fields.
- **Effective name stays in `name`.** `name` always returns the best available
  name (`authoritative_name` if set, else `auto_name`). Consumers that don't
  care about naming source continue to work unchanged.
- **`source_filter` on `ListServicesRequest`** allows callers to request only
  auto-observed or only watched services without client-side filtering.

**Benefits:**

- A single RPC surfaces the complete picture of what the agent is observing.
- CLI (RFD 107) and MCP tools can render auto vs. watched services differently
  without parsing the name.
- Future adaptors (Kubernetes, Docker) surface their source via
  `SERVICE_NAMING_SOURCE_ADAPTOR` without a new field.

### Component Changes

1. **`proto/coral/agent/v1/agent.proto`**:
   - Add `ServiceNamingSource` enum.
   - Extend `ServiceStatus` with new fields (see API Changes).
   - Add `source_filter` to `ListServicesRequest`.
   - Regenerate Go bindings.

2. **`ListServices` RPC handler** (`internal/agent/`):
   - Iterate `agent.services map[int32]*ServiceEntry` (RFD 104).
   - Populate all new fields from `ServiceEntry`.
   - Apply `source_filter` when set.
   - Remove the old `monitors`-only iteration.

## API Changes

### Protobuf Definitions

```protobuf
// ServiceNamingSource indicates how the effective service name was determined.
enum ServiceNamingSource {
    SERVICE_NAMING_SOURCE_UNSPECIFIED = 0;

    // Name generated by the ServiceNameAdaptor chain (process binary name
    // or port-based fallback).
    SERVICE_NAMING_SOURCE_AUTO = 1;

    // Name explicitly set by the operator via `coral services watch`.
    SERVICE_NAMING_SOURCE_AUTHORITATIVE = 2;

    // Name resolved by an external adaptor (e.g., Kubernetes pod labels).
    // Reserved for future adaptor implementations.
    SERVICE_NAMING_SOURCE_ADAPTOR = 3;
}

// Extended ServiceStatus
message ServiceStatus {
    string name = 1;              // Effective: authoritative_name ?? auto_name
    int32  port = 2;
    string health_endpoint = 3;   // Only set for watched services
    string service_type = 4;
    map<string, string> labels = 5;
    string status = 6;            // healthy/unhealthy/unknown; empty for Tier 0
    google.protobuf.Timestamp last_check = 7;
    string error = 8;
    int32  process_id = 9;
    string binary_path = 10;
    string binary_hash = 11;

    // New fields
    string auto_name = 12;                     // Name from adaptor chain
    string authoritative_name = 13;            // Name from coral services watch
    ServiceNamingSource naming_source = 14;
    bool   has_monitor = 15;                   // True if ServiceMonitor running (Tier 1+)
    uint32 observation_tier = 16;              // Highest active tier (0, 1, 2, ...)
}

// Extended ListServicesRequest
message ListServicesRequest {
    // Filter by naming source. Unset returns all services.
    optional ServiceNamingSource source_filter = 1;
}
```

## Testing Strategy

### Unit Tests

See Phase 3 above.

### Integration Tests

- Start agent; send OTLP for port 3000 (`node`) and port 6379 (`redis`).
- Call `ListServices` (no filter): assert two entries, both
  `naming_source = AUTO`, both `observation_tier = 0`, `has_monitor = false`.
- Call `ConnectService` for port 3000 with name `frontend` and health
  endpoint.
- Call `ListServices` again: assert port 3000 has
  `naming_source = AUTHORITATIVE`, `authoritative_name = "frontend"`,
  `has_monitor = true`, `observation_tier = 1`.
- Call `ListServices` with `source_filter = AUTO`: assert only port 6379
  returned.
